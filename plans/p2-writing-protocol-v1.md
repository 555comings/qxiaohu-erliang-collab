# P2 写入协议 v1

制定对象：二两的记忆系统

适用范围：`MEMORY.md`、`memory/YYYY-MM-DD.md`、以及 `do-not-write` 排除桶

版本目标：把“什么必须写、写到哪、怎么标状态、什么时候升级/降级”变成可直接执行的规则，而不是靠感觉。

---

## 0. 一句话原则

先判断值不值得记，再判断记到哪；没有长期价值的东西不要硬塞进 `MEMORY.md`，没有行动价值的噪音不要写进任何记忆。

---

## 1. 三分法：写到哪里

| 桶 | 位置 | 写什么 | 不写什么 |
| --- | --- | --- | --- |
| 长期记忆 | `MEMORY.md` | 稳定事实、长期偏好、已验证规则、持续中的长期项目状态 | 当天流水、单次任务细节、未经验证的猜测 |
| 每日记忆 | `memory/YYYY-MM-DD.md` | 当天任务推进、卡点、决策、返工、交接、刚发现但还未沉淀的规则 | 与后续无关的聊天废话、重复抄写长期记忆 |
| 排除桶 | `do-not-write`（概念桶，不落盘） | 不写入任何记忆；只留在当前任务文件或直接忽略 | 无来源猜测、敏感但无必要信息、重复噪音、一次性情绪宣泄 |

执行时只允许三选一，不能既写 daily 又把同一段原样抄进 `MEMORY.md`。

---

## 2. 决策树：30 秒内决定写哪里

按顺序判断：

1. 这条信息以后还会影响判断、协作或执行吗？
   - 否：进 `do-not-write`
   - 是：继续
2. 这条信息是否只对今天或当前任务有用？
   - 是：写 `memory/YYYY-MM-DD.md`
   - 否：继续
3. 这条信息是否已经验证，且预计一周后仍然有用？
   - 是：写 `MEMORY.md`
   - 否：先写 `memory/YYYY-MM-DD.md`，状态标 `draft`

默认落点是 daily，不是 `MEMORY.md`。

---

## 3. 强制写入触发器

以下情况发生后，必须写一条，不能只“记在脑子里”：

### 3.1 立即写（10 分钟内）

1. 用户明确说“记住这个”或等价表达。
2. 形成了明确决策，且后续任务要按这个决策执行。
3. 发现一个会重复出现的系统性错误。
4. 出现返工，且返工原因会影响之后的交接方式。
5. 新确认了一条用户偏好、协作规则或环境事实。

### 3.2 收口前必须写

1. 一个任务被声明为 `done` 前。
2. 一个任务被交接给别人审核前。
3. 一个任务被暂停、阻塞、放弃前。
4. 一条 `draft` 规则被提升为 `verified` 前，要先在当天 daily 留证据。

### 3.3 不满足触发器时

可以不写。协议不是要求把每一句话都存档，而是要求关键状态变化不遗漏。

---

## 4. 状态标记与升级规则

所有会被未来再次引用的条目，都用以下状态之一：

- `[draft]`：刚记下，只有单次观察、单一来源，或还没被后续任务验证。
- `[verified]`：已被用户明确确认，或被至少两次独立任务/场景验证。
- `[stale]`：曾经有效，但因为时间、环境、流程变化，可能已经不可靠。
- `[deprecated]`：已被新的规则/事实明确替代，保留只为追溯。

### 4.1 状态切换规则

- `draft -> verified`
  - 满足任一条件：
    1. 用户明确确认；
    2. 同一规则在两次独立任务里都成立；
    3. 相关执行结果证明它可靠。
- `verified -> stale`
  - 满足任一条件：
    1. 30 天以上未再使用，且环境已变化；
    2. 出现与旧规则冲突的新迹象，但还没完全确认替代关系。
- `stale -> verified`
  - 重新验证通过。
- `verified/stale -> deprecated`
  - 新规则已经明确接管，旧规则不能再作为默认依据。

### 4.2 状态和位置的关系

- `draft` 默认写 daily。
- `verified` 可以留在 daily，也可以被提升到 `MEMORY.md`。
- `stale` 和 `deprecated` 如果已经存在于 `MEMORY.md`，就在原条目上改状态，不要静默删除。

---

## 5. 提升规则：什么时候从 daily 升到 `MEMORY.md`

只有同时满足下面 3 条，才允许提升：

1. 不是一次性信息；
2. 状态已经是 `[verified]`；
3. 对未来一周以上的判断、协作或执行仍然有帮助。

常见可提升内容：

- 用户长期偏好
- 固定协作规则
- 长期项目的当前阶段
- 已证实的工具/环境限制
- 明确的角色分工

不满足 3 条时，继续留在 daily。

---

## 6. `do-not-write` 清单

命中以下任一项，直接不写入任何记忆：

1. 没有来源、没有证据的猜测。
2. 只在当前几分钟内有用的琐碎中间态。
3. 已经写过、且没有新增信息的重复内容。
4. 纯情绪化吐槽、无后续行动价值的反应。
5. 敏感信息，但记下来对后续任务没有必要。
6. 可以在任务产物里找到、没必要再复制进记忆的原始细节。

遇到边界情况时，优先少写，宁可先放 daily 的 `[draft]`，也不要把未验证信息写进 `MEMORY.md`。

---

## 7. 最小模板

### 7.1 daily 最小模板

写入 `memory/YYYY-MM-DD.md`：

```md
## [memory] 2026-03-13 14:40
- Trigger: <done | rework | decision | blocked | remember-request | lesson>
- Scope: <task or topic>
- Entry: [draft|verified|stale|deprecated] <一句话事实/规则/状态>
- Why: <为什么值得记>
- Next: <是否需要验证、提升到 MEMORY.md、或等待下一次观察>
```

要求：每条尽量 5 行内解决，避免写成长散文。

### 7.2 `MEMORY.md` 最小模板

```md
- [verified] <主题>：<稳定事实或规则>。
  Evidence: <用户确认 / 两次任务验证 / 具体来源>
  Updated: 2026-03-13
```

如果状态不是 `verified`，原则上不要新建到 `MEMORY.md`。

### 7.3 旧规则改状态模板

```md
- [deprecated] <旧规则>。
  Replaced by: <新规则>
  Updated: 2026-03-28
```

```md
- [stale] <旧事实或旧规则>。
  Check next: <需要重新确认什么>
  Updated: 2026-03-28
```

---

## 8. 执行步骤

每次写入都按这个顺序：

1. 识别是否命中强制触发器。
2. 先分桶：`MEMORY.md` / daily / `do-not-write`。
3. 给条目加状态，不允许裸写关键事实。
4. 用最小模板落盘，优先短句。
5. 如果是 `draft`，在 `Next` 里写清下一次验证条件。
6. 如果是提升到 `MEMORY.md`，保留 daily 里的原始验证痕迹，不做原文复制，只写更短的稳定版。

---

## 9. Worked Example：一次真实协作规则如何落盘

场景：二两在审核 `execution-checklist-v2` 的返工时发现，执行者没有同步预计交付时间，导致审核方不知道该等多久。这个问题后来又在后续任务中被再次证实，因此它从 daily 的 `draft` 升级成 `MEMORY.md` 里的 `verified` 规则，最后在流程自动化后被 `deprecated`。

### 9.1 第一次发现：先写 daily，状态是 `draft`

文件：`memory/2026-03-13.md`

```md
## [memory] 2026-03-13 16:20
- Trigger: rework
- Scope: execution-checklist-v2 review loop
- Entry: [draft] 进入返工时，执行者需要同步预计交付时间，否则审核方无法判断等待时长。
- Why: 这次返工已经因此出现等待和催办，后续协作会重复受影响。
- Next: 如果用户或下一次任务再次确认，就升级为 verified。
```

解释：这时它只是一次观察，还不能直接进 `MEMORY.md`。

### 9.2 第二次验证：仍在 daily 留证据，但状态改成 `verified`

文件：`memory/2026-03-18.md`

```md
## [memory] 2026-03-18 11:05
- Trigger: lesson
- Scope: second rework handoff
- Entry: [verified] 进入返工时必须同步预计交付时间；这条规则再次成立，并减少了追问成本。
- Why: 已在两次独立返工场景中成立，可以作为默认协作规则。
- Next: 提升到 MEMORY.md，作为长期协作规则保存。
```

### 9.3 提升到长期记忆：写稳定版，不复制流水

文件：`MEMORY.md`

```md
- [verified] 协作规则：进入返工时，执行者必须同步预计交付时间。
  Evidence: 在两次独立返工任务中成立，并已在阶段三复盘中明确提出。
  Updated: 2026-03-18
```

解释：`MEMORY.md` 只保留规则本身和证据来源，不抄整段任务经过。

### 9.4 后续流程变化：先标 `stale`，确认替代后再标 `deprecated`

如果之后启用了自动状态面板，返工 ETA 由系统自动生成，旧规则先变成可能过期：

```md
- [stale] 协作规则：进入返工时，执行者必须同步预计交付时间。
  Check next: 确认自动状态面板是否已稳定覆盖所有返工任务。
  Updated: 2026-04-02
```

等自动状态面板稳定后，再改成：

```md
- [deprecated] 协作规则：进入返工时，执行者必须同步预计交付时间。
  Replaced by: 返工 ETA 由自动状态面板统一生成，人工只需确认是否接受。
  Updated: 2026-04-10
```

这个例子展示了完整链路：`draft -> verified -> stale -> deprecated`。

---

## 10. 审计要求

每次回顾时，只检查 4 件事：

1. 最近 3 个任务是否都命中了该写就写。
2. `MEMORY.md` 里是否混进了当天流水账。
3. daily 里是否有长期有价值但还没提升的 `[verified]` 条目。
4. 是否存在已经失效但还没标 `stale/deprecated` 的旧规则。

---

## 11. 通过标准

满足以下条件，就算 `P2 写入协议 v1` 已经落地：

- 最近 3 个任务都能在对应 daily 找到触发记录。
- `MEMORY.md` 中新增条目全部为 `[verified]`。
- 同一事实不再被原样同时抄进 daily 和 `MEMORY.md`。
- 至少有 1 条条目经历过 `draft -> verified` 的升级。
- 出现过期规则时，不删除，改为 `stale` 或 `deprecated`。

---

## 12. 默认判断顺序

拿不准时，统一按这个顺序：

1. 先问：以后还会用到吗？
2. 再问：只对今天有用，还是会跨天复用？
3. 再问：已经验证了吗？
4. 结论：默认写 daily；只有稳定、已验证、跨天有用的内容才进 `MEMORY.md`。

这份协议的核心不是“多写”，而是“写对地方、带上状态、能被以后的人看懂并复用”。
